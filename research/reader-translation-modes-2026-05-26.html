The flow

1
See foreign text only. No English on screen. The reader tries the sentence under its own steam.
2
Tap words. Each tap cycles red → yellow → unmark. Same tap-cycle as sentence-review. Each tap is a server POST /api/words/mark live.
3
Reveal (optional). Slides the English under each foreign sentence, italic, indented. Toggle off to go back to immersion. No commit yet — this is purely viewing.
4
Next. Page sweep fires: marked words count as misses (red = unknown, yellow = encountered); every unmarked word in the page counts as known at FSRS rating 3. Idempotent per page via client_review_id = pr:<story>:<n>. Advances.
5
Prev. Re-reads the previous page. Per-word marks restore from cache; you can edit them and the server updates live. The sweep does not re-fire (same client_review_id).

What changes vs. PR #149

  • "Show English" button → "Reveal". Pure rename.
  • Reveal layout swaps from full-page English block (mode A) to sentence-by-sentence interleaved (mode B). This is the substantive change.
  • Backend prerequisite: sentence-level English doesn't exist yet — only page-level pages.translation_en. Need a new sentences.translation_en populated by the same lazy call. (See "implementation" below.)
  • Tap-to-mark, Prev / Next, idempotent sweep, page cache: already shipped in PR #149. No change.

Implementation: where the sentence English comes from

The cleanest path is one extra column on sentences, populated lazily by extending ensure_page_translation to return per-sentence pairs in a single LLM call (JSON list of {sentence_id, en}). Same write-lock-safe pattern as the page version. Frontend reads page.sentences[].translation_en instead of page.translation_en.

Cost of choosing this over splitting the page-level string client-side: one extra column, one JSON-schema constrained prompt. Cost of not choosing it: silent misalignment when a Latin sentence breaks into two English sentences (or vice versa) — common with periodic Latin and very real with Eutropius's chapter 1 above.
