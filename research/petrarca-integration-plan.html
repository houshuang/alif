1. Hypothesis Annotation Integration Do this

Why this is the highest-value integration

Hypothesis annotations are user-curated atomic claims — far more reliable than LLM-extracted claims because the user selected the exact passage they found noteworthy. This is the strongest knowledge signal Petrarca can get, and it's currently ignored.

Your Hypothesis account already exists (3 annotations from 2021). The API is open, unthrottled, and well-documented. Incremental sync via search_after cursor is trivial.

Effort: ~370 lines, 1-2 sessions Dependencies: None (just requests) Env: HYPOTHESIS_API_KEY, HYPOTHESIS_USERNAME

Data model

Each Hypothesis annotation contains:

FieldSourcePetrarca use
target.selector[TextQuoteSelector].exactThe highlighted passageUser-curated atomic claim
textUser's note/commentElaboration, interpretation signal
tags[]User-assigned tagsMap to entities / curriculum nodes
uriPage URLMatch to existing articles or auto-import
createdTimestampReading activity timeline

Four integration points

A. Highlighted passages → atomic claims (highest value)

When a user highlights text, that's an atomic claim with claim_type='user_highlight'. These claims should get higher weight in novelty detection — they represent what the user found noteworthy, not what an LLM thought was important. User notes (text field) become claim annotations.

B. Reading depth signal

Annotation count per article = reading depth. An article with 12 highlights was deeply read; one with 0 was skimmed. This feeds into feed ranking (don't re-surface deeply-read articles) and knowledge state confidence.

C. Tag → entity / curriculum linking

If a user tags a highlight with "Byzantine Empire", match to shared_entities and upgrade linked curriculum nodes to engaged. This is direct user-declared knowledge evidence.

D. Auto-import annotated articles

If the user annotates a URL not in Petrarca's database, auto-import it via the existing article pipeline. Annotated articles are high-priority candidates — the user already cared enough to highlight.

Sync implementation

# Incremental sync using search_after cursor (gooseberry pattern)
def sync_annotations(username: str, api_key: str, since: str | None = None):
    params = {
        "user": f"acct:{username}@hypothes.is",
        "sort": "updated", "order": "asc", "limit": 200,
    }
    if since:
        params["search_after"] = since

    while True:
        resp = requests.get("https://hypothes.is/api/search",
                           headers={"Authorization": f"Bearer {api_key}"},
                           params=params)
        batch = resp.json()["rows"]
        if not batch:
            break
        yield from batch
        params["search_after"] = batch[-1]["updated"]

Pipeline integration

Add as Step 0c in the 4-hour cron, after Readwise Reader fetch:

# In build_articles.py cron:

2. Entity Extraction Prompt Engineering Do this

What to adopt from Graham's repos

Three specific prompt engineering patterns from steamroller, kg-hybrid, and claude_antiquities_extractor_skill that would improve Petrarca's entity extraction with zero new dependencies:

Effort: Prompt changes only, 1 session Dependencies: None Risk: Low — A/B test against current extraction

Change 1: Constrained relationship types

Currently Petrarca asks for "related entity pairs (max 5 per entity)" with free-form relationship descriptions. Graham's repos constrain to a fixed vocabulary, producing more structured, queryable graphs.

Proposed relationship vocabulary for Petrarca's historical domains (~15 types):

RULED, CONQUERED, FOUNDED, INFLUENCED, SUCCEEDED,
PART_OF, LOCATED_IN, CONTEMPORARY_OF, ALLIED_WITH,
OPPOSED, CAUSED, RESULTED_IN, DERIVED_FROM,
CREATED, STUDIED_BY

Each predicate has a directional definition (e.g. RULED: subject governed/controlled object). The LLM selects from this list rather than inventing relationship descriptions.

Change 2: Canonical ID convention

Instead of matching entities by case-insensitive name (fragile for "Alexander" vs "Alexander the Great" vs "Alexander III of Macedon"), assign canonical IDs: 

PERSON:       firstname_lastname       (e.g. roger_ii_sicily)
PLACE:        place_name_geography     (e.g. palermo_sicily)
EVENT:        short_description_date   (e.g. norman_conquest_sicily_1061)
CONCEPT:      descriptive_slug         (e.g. convivencia)
ORGANIZATION: abbreviated_name        (e.g. fatimid_caliphate)

These IDs are stable across articles. When two articles both mention roger_ii_sicily, they link automatically without fuzzy matching.

Change 3: Mentions array

Track all text variants encountered for each entity. Currently Petrarca has {name, aliases}. Upgrade to a comprehensive mentions array populated from actual article text: 

{
  "canonical_id": "roger_ii_sicily",
  "name": "Roger II of Sicily",
  "mentions": ["Roger II", "Roger", "the Norman king", "King Roger"],
  "type": "PERSON",
  "relationships": [
    {"target": "kingdom_of_sicily", "type": "RULED"},
    {"target": "fatimid_caliphate", "type": "OPPOSED"}
  ]
}

Change 4: Prescribed extraction workflow (from the Claude skill)

Add a step-by-step procedure to the extraction prompt instead of a single "extract entities" instruction: 

1. Read the article thoroughly
2. Identify all named entities (people, places, events, concepts, organizations)
3. Assign canonical_id to each using the convention above
4. Collect all text mentions of each entity
5. Record attributes (dates, type, description)
6. Identify relationships using ONLY the predicate vocabulary
7. Check: did you merge variants that refer to the same entity?

How to test

Run through the pipeline-eval framework: create a fixture with 5 articles from different domains, extract entities with current prompt and new prompt, compare coverage and consistency. Key metric: do the same entities get the same canonical_id across articles?

Files to modify: scripts/extract_entity_concepts.py (prompt changes), scripts/db.py (add canonical_id, mentions, relationship_type columns)

3. Zotero Translation Server Worth trying

The pitch

Zotero Translation Server is a standalone Docker service with 600+ community-maintained site-specific extractors. It complements trafilatura — Zotero does metadata only, trafilatura does content. Running both gives you richer article records.

Effort: ~100 lines + Docker setup Cost: ~500-700MB image, ~200MB RAM Latency: 1-5s per URL (parallel with trafilatura)

What Zotero gives you that trafilatura doesn't

FieldValue for Petrarca
itemTypeSemantic classification: blogPost vs newspaperArticle vs journalArticle. Much richer than hostname-based content_type.
creators[]Structured first/last name + role (author vs editor vs translator). Better than raw author string.
abstractNoteArticle abstract from meta tags. Free summary without LLM cost.
DOI / PMIDUnique identifiers for academic articles. Enables CrossRef/PubMed enrichment via POST /search.
publicationTitlePublication name (journal, newspaper, blog name). More specific than hostname.
languageISO language code from meta tags.
tags[]Author-assigned keywords from meta tags. Free signal for entity matching.

Architecture

# docker-compose.yml addition:
services:
  zotero:
    image: zotero/translation-server
    ports: ["1969:1969"]
    restart: unless-stopped

# Python client (~20 lines):
def zotero_metadata(url: str) -> dict | None:
    try:
        resp = requests.post("http://localhost:1969/web",
                            data=url,
                            headers={"Content-Type": "text/plain"},
                            timeout=15)
        if resp.status_code == 200:
            items = resp.json()
            return items[0] if items else None
    except Exception:
        return None  # graceful degradation

Integration pattern

Run Zotero and trafilatura in parallel for each URL. Merge results:

  • Content (markdown body): always from trafilatura
  • Title: prefer Zotero (site-specific extractors are smarter) with trafilatura fallback
  • Author: prefer Zotero (structured name parsing)
  • New fields: item_type, abstract, doi, publication_name, language, meta_tags

Notable limitation

Zotero's Embedded Metadata translator does not parse JSON-LD (<script type="application/ld+json">). Many modern sites use JSON-LD exclusively. PressForward's client-side parser does handle JSON-LD, but it requires running in a browser context. For server-side processing, this is a gap.

Recommendation

Try it on 50 articles from different sources and measure how many fields Zotero fills that trafilatura doesn't. If the hit rate is >30%, deploy as a sidecar. If most articles already have good metadata from trafilatura, skip it.

Files to create/modify: docker-compose.yml (add service), scripts/build_articles.py (parallel metadata fetch + merge), scripts/db.py (add metadata columns)

4. Discourser-style Semantic Vectors Research spike

The idea

Discourser defines custom semantic dimensions by providing positive and negative term sets, then projects documents onto them. For Petrarca, this could add continuous semantic coordinates to articles beyond discrete topic tags.

Example dimensions for Petrarca's historical domains

# Each dimension defined by positive/negative term sets:
"primary_vs_secondary":
  pos: ["chronicle", "inscription", "manuscript", "testimony", "decree"]
  neg: ["argues", "interprets", "historiography", "scholarship", "analysis"]

"military_vs_cultural":
  pos: ["conquest", "siege", "army", "battle", "fortification"]
  neg: ["art", "poetry", "mosque", "cathedral", "translation"]

"technical_vs_narrative":
  pos: ["data", "methodology", "analysis", "evidence", "quantitative"]
  neg: ["story", "journey", "experience", "vivid", "dramatic"]

Articles get projected onto these axes, giving coordinates like {primary_vs_secondary: 0.73, military_vs_cultural: -0.2, technical_vs_narrative: 0.45}. This enriches feed ranking and could help the novelty system distinguish "same topic, different angle" from "same topic, same angle."

Three methods from discourser

MethodHowQuality
Orthogonal projectionpos_centroid - neg_centroid, remove shared componentBest separation
Weighted differenceSimple centroid subtractionFast, decent
PCA axisMax variance axis between pos/neg embeddingsData-driven

Why this is a research spike, not an integration

Petrarca already has MiniLM embeddings via limbic. The computation is trivial. The hard part is defining the right dimensions — this requires experimentation to find axes that actually discriminate usefully between articles. A spike would:

  1. Define 3-5 candidate dimensions with term sets
  2. Project 100 existing articles onto them
  3. Check: do the projections correlate with meaningful differences?
  4. If yes, add to article metadata and feed ranking
Spike effort: ~2 hours, Jupyter notebook Full integration: ~150 lines if spike succeeds Dependencies: Already has MiniLM via limbic

5. Dual-Model Verification Consider later

What steamroller does

After initial entity/triplet extraction, two additional LLMs review the output: Model 1 checks triplets against source text and suggests corrections, Model 2 reviews Model 1's critique and produces final output. This catches hallucinated entities and wrong relationships.

Adaptation for Petrarca

After entity extraction (Gemini Flash), run a verification pass (Claude Haiku via claude -p, free): 

# Verification prompt:
"Here is an article and the entities extracted from it.
For each entity, verify:
1. Is it actually mentioned in the article? (not hallucinated)
2. Is the canonical_id consistent with other articles?
3. Are the relationships supported by the text?
Flag any issues."

This is cheap (Haiku, free via CLI) and catches the biggest failure mode: hallucinated entities that pollute the knowledge graph. But it doubles the LLM calls per article.

When to do this

Wait until after the prompt engineering changes (#2 above). Constrained relationship types and canonical IDs may reduce hallucination enough that verification isn't needed. If entity quality is still an issue after those changes, add verification.

Dropped: PressForward Metadata Cascade Skip

Why not

Deep-dive confirmed that trafilatura already covers everything PressForward does for content extraction, and does it better (trafilatura's algorithm outperforms standalone Readability on edge cases). PressForward's metadata cascade (OG → Twitter → JSON-LD → meta description) is well-structured but trafilatura handles the same sources.

The interesting part of PressForward is its editorial workflow (feed → nomination → publication), but Petrarca already has its own article lifecycle. Not worth the WordPress/PHP dependency.

One pattern worth stealing (no integration needed): the image fallback chain (OG image → link[rel=image_src] → Twitter image → first large <img>). Add to trafilatura post-processing if Petrarca needs featured images.
