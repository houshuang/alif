Spike report · 2026-04-22

Bookify renderer choice — paged.js rejected, WeasyPrint stays

A 1-hour spike to test whether paged.js is a viable replacement for WeasyPrint as the per-page-footnote renderer in bookify_arabic.py. Result: no, and the WeasyPrint footnote path that was assumed broken is actually fine.

Decision: Keep WeasyPrint. Generate the missing kalila_dove.footnotes.pdf with the existing --format footnotes. Do not switch to paged.js (or any Chromium-based pager) without first restructuring the HTML to break long paragraphs into per-sentence <p> blocks — and even then it gains nothing at our scale.

Why we ran the spike

The Kalila reader had two PDF variants — bilingual.pdf (sentence pairs, A4 landscape) and glossary.pdf (preface vocab + underlined Arabic-only body, A5). The user wanted a third: per-page footnotes for first-occurrence glosses inline. IDEAS.md claimed:

WeasyPrint chokes on 600+ floating footnotes. Switch to Paged.js (Chromium headless).

So we'd never actually generated the footnote PDF. The spike: install pagedjs-cli alongside WeasyPrint, render the same Kalila HTML through both, and pick a winner.

Setup

  • WeasyPrint: 67.0 (already installed in system Python).
  • paged.js: pagedjs-cli 0.4.4 + bundled puppeteer 20.9 + Chrome-for-Testing 115. Installed under /tmp/claude/pagedjs/.
  • Source: backend/data/kalila_dove.json — 5 paragraphs, 3162 tokens, 432 distinct unfamiliar lemmas.
  • HTML variants tested: full footnote (479 anchors), top-N gated (N ∈ {25, 50, 100, 200, 300, 432}), no-footnote baseline, bilingual.

Local quirk worth remembering

pagedjs-cli on this Mac requires --browserArgs "--no-sandbox,--disable-setuid-sandbox". Without those, Chrome starts but never prints its WebSocket endpoint URL to stdout — puppeteer times out at 30s with the unhelpful TimeoutError: Timed out after 30000 ms while waiting for the WS endpoint URL to appear in stdout! error. Likely caused by Chrome 115 x64 running under Rosetta on Apple Silicon (puppeteer's bundled Chrome download is x64 only).

WeasyPrint scaling is super-linear (~O(n·log n)) but well-behaved out to 432 footnotes. The SLOW >100 lemmas warning was a defensive guess, not measured.

What paged.js failure looks like

Every paged.js run produced the same skeleton: page 1 = title, page 2 = preface vocab (full), page 3 = ~3 lines of body text + footnotes (when present), page 4 = colophon. The other ~25-33 pages of body content vanished silently. Exit code 0. No warning.

This held even with footnotes stripped entirely. So it's not a footnote bug — paged.js's content-flow handler can't satisfy break-inside: avoid on our too-tall blocks (each Kalila "paragraph" is a 500-800-token chapter section with no internal break) and gives up rather than forcing a break.

This is the scariest class of renderer bug: it "succeeds" but lies about what it rendered. If we'd shipped on paged.js without diffing page counts, the user would have ended up with a pretty-looking 4-page chapter instead of a 37-page one, with no error to investigate.

Fix path for paged.js (not pursued)

Restructure the ingest to split each paragraph into per-sentence <p> blocks (the bilingual format already does this with sentence-pair table rows; the body format does not). With sentence-sized blocks, paged.js's break-finder has natural anchors to work with. But this gains us nothing — WeasyPrint already handles the existing structure in 30 s — so we're not going to do it.

What we shipped from this spike

  1. The actual artifact: backend/data/kalila_dove.footnotes.pdf (37 A5 pages, 432 first-occurrence footnotes, ~12 footnotes/page, beautiful Scheherazade shaping). Generated via python3 scripts/bookify_arabic.py --db-path data/alif.prod.db render data/kalila_dove.json data/kalila_dove.footnotes.pdf --format footnotes.
  2. IDEAS.md: marked the per-page-footnote follow-up [DONE 2026-04-22], struck the paged.js suggestion.
  3. bookify_arabic.py: removed the SLOW >100 lemmas warning from docstring and --help; added a "do not switch to paged.js" note on the footnotes branch.
  4. research/experiment-log.md: new entry at top.
  5. Memory project_bookify_arabic.md: settled the renderer choice with full reasoning, so we don't re-spike paged.js next session.

Reproducing the spike

If a future session wants to re-test (e.g. after a paged.js major release or a chapter with a different paragraph structure):

# 1. install pagedjs-cli (needs sandbox off for npm cache)
mkdir -p /tmp/claude/pagedjs && cd /tmp/claude/pagedjs
npm init -y && npm install pagedjs-cli
node node_modules/puppeteer/install.js   # fetches Chromium ~150MB

# 2. dump the footnote HTML without running WeasyPrint
cd /Users/stian/src/alif/backend && python3 -c "
import json, sys
sys.path.insert(0, '.')
from scripts.bookify_arabic import render_html
aligned = json.load(open('data/kalila_dove.json'))
open('/tmp/claude/test.html', 'w').write(render_html(aligned, fmt='footnotes'))
"

# 3. render with both, compare page counts
time python3 -c "from weasyprint import HTML; HTML(filename='/tmp/claude/test.html').write_pdf('/tmp/claude/weasy.pdf')"
time /tmp/claude/pagedjs/node_modules/.bin/pagedjs-cli \
  -i /tmp/claude/test.html -o /tmp/claude/pagedjs.pdf \
  --browserArgs "--no-sandbox,--disable-setuid-sandbox"

pdfinfo /tmp/claude/weasy.pdf | grep Pages
pdfinfo /tmp/claude/pagedjs.pdf | grep Pages
# If pagedjs page count is dramatically lower than weasy, paged.js is still
# silently truncating — keep WeasyPrint.
