What didn't work

silent failure timeout trap premature delegation

The sandbox / claude -p interaction

This cost the most time. The harness's default sandbox blocks writes to /Users/stian/.claude.json, which the claude CLI needs to update credentials every call. Symptom: every generate_completion(cli_only=True) call fails with Claude CLI failed for claude_sonnet and cli_only=True, independent of prompt length or model. Fix: pass dangerouslyDisableSandbox: true when the command chain ends up invoking claude -p.

Tell: all Claude CLI calls fail at the same speed, with identical generic error message, regardless of input. If you see that pattern, run echo hi | timeout 30 claude -p --output-format text as a direct probe — the real error (EPERM open '/Users/stian/.claude.json') shows there.

Prompt-length-driven timeouts

Sonnet via CLI has a 240s timeout in llm.py. A 2.5K-char classical-Arabic paragraph produces ~5K chars of vocalized output, and Sonnet crosses 240s on all five. Smaller batches (8 sentences) help but still intermittent, and Haiku per-batch costs 2-3 min walltime — 34 batches is not worth the wait when I can do it directly.

Batched translation (translate_paragraphs as originally written)

The original code bundled all six paragraphs into a single Sonnet call — ~17K chars in, ~17K out. Any failure nuked all six; often happened. Rewrote to per-paragraph calls (smaller, retryable, partial-success tolerant). This had already been a pending fix from the prior session.

Delegating tasks I can do directly

Repeatedly reached for claude -p subprocesses for vocalization and alignment when I am Claude, in context, with the text in front of me. The correct move: read the paragraph, output the vocalized paragraph to a file in the same turn. 5 paragraphs took under 5 minutes and zero failures.

Monitor + tail -f buffering

A Monitor watching tail -f on an in-progress log can stream fine if the upstream writer is line-buffered, but Python's default stdout buffering in a piped context needs python3 -u. Earlier watches hung because stdout never flushed.

What worked

direct output bundled font idempotent import sentence-pair layout

Bundling the font

Scheherazade New is installed only in backend/data/fonts/ScheherazadeNew/ and referenced via @font-face { src: url(file://…); }. No dependency on system fonts, no network fetch. Renders identically on local dev and server. The WeasyPrint docs warn against http:// @font-face — it can hang for minutes waiting for a fetch. Local file URLs don't.

Two-tier highlighting

Showing every unfamiliar word with the same treatment is noise (479 words in this chapter). Showing only preface words leaves 400 non-preface words invisible. The fix: two visually distinct styles. Saffron solid underline (.tok.new) for preface entries you can look up; faint gray dotted underline (.tok.new-dim) for words to infer. Legend on the preface page.

Per-paragraph, per-sentence-pair layout for bilingual

The old bilingual put a whole paragraph AR on the right and a whole paragraph EN on the left — for a 4K-char paragraph, one side or the other was blank for half the page. Splitting into pairs: [{ar, en}, …] (sentence-level) and rendering each pair as a single table row with break-inside: avoid keeps both languages visible on every page. 177 pairs across the chapter — no wasted space.

Idempotent introduce subcommand

bookify_arabic.py introduce <json> --top 25 [--dry-run]. Runs the full Alif-lemma resolution chain (exact vocalized match → bare-form match → resolve_existing_lemma → new Lemma with source='scaffold' + run_quality_gates), then seeds UserLemmaKnowledge rows with knowledge_state='encountered', source='book'. Safe to re-run; dry-run previews.

Lessons for future sessions

  1. Before reaching for claude -p in a subprocess: ask, "Can I just write this directly?" For vocalization, translation, sentence alignment, or any text task where I have the input in context, the answer is nearly always yes. Subprocess-CLI is for when you're in a script the user is running standalone — not for in-session work.
  2. If you must use CLI in session: start with dangerouslyDisableSandbox: true and a direct-probe call (echo … | claude -p) to confirm the CLI itself works before chaining.
  3. Prompt-length scaling: Sonnet CLI timeout is 240s. Classical Arabic vocalization roughly doubles char count. Keep single calls under ~2K chars output.
  4. Preview before committing long output. Render 1-2 PDF pages to PNG (pdftoppm -png -r 200 -f 1 -l 2 <pdf> …) and inspect before declaring done. Several iterations caught layout bugs this way (A5-on-bilingual, missing highlights, preface overflow).
  5. Two-tier highlighting + legend is the right default for vocab-assisted reading. Single-tier is noise; preface-only is uninformative.
  6. For corpus operations on prod DB: backup → dry-run → run → log via scripts/log_activity.py. Every time.
  7. Don't commit stuck processes in background. A long-running Monitor on a buffered stdout will hang silently. Use python3 -u and short-interval flushing in the script.
  8. Use --dry-run for import/mutation commands and print the resolved plan before committing.

How to do this faster next time

For a new chapter, the end-to-end path is:

  1. Pull fresh prod DB to pick up new lemmas/ULK from recent reviews: 
    scp alif:/opt/alif/backend/data/alif.db backend/data/alif.prod.db
  2. Clean + vocalize. Don't batch out to CLI; read the chapter, write the vocalized file directly. For 5 paragraphs of classical prose this is ~5 min of output.
  3. Ingest with gloss, without translate/align (those will be done in-session): 
    cd backend
python3 scripts/bookify_arabic.py --db-path data/alif.prod.db ingest \
  data/{book}.vocalized.txt data/{book}.json \
  --title "…" --author "…" --no-translate
  4. Align (in-session, one Write per paragraph to /tmp/claude/pairs_p{n}.json), then patch: 
    python3 /tmp/claude/patch_pairs.py
  5. Render both formats (fast, ~15 s each): 
    python3 scripts/bookify_arabic.py --db-path data/alif.prod.db render \
  data/{book}.json data/{book}.bilingual.pdf --format bilingual
python3 scripts/bookify_arabic.py --db-path data/alif.prod.db render \
  data/{book}.json data/{book}.glossary.pdf --format glossary
  6. Preview before declaring done: 
    pdftoppm -png -r 200 -f 1 -l 4 data/{book}.bilingual.pdf /tmp/claude/preview
# Read the PNGs, spot issues, iterate.
  7. Introduce top-N into Alif (dry-run first, backup prod, run real, log): 
    # dry run
python3 scripts/bookify_arabic.py --db-path data/alif.prod.db introduce \
  data/{book}.json --top 25 --dry-run

# prod
ssh alif "cp /opt/alif/backend/data/alif.db /opt/alif-backups/alif_pre_bookify_$(date +%Y%m%d_%H%M%S).db"
scp scripts/bookify_arabic.py alif:/tmp/
scp data/{book}.json alif:/tmp/
ssh alif "cd /opt/alif/backend && PYTHONPATH=/opt/limbic .venv/bin/python3 \
  /tmp/bookify_arabic.py introduce /tmp/{book}.json --top 25"
ssh alif "cd /opt/alif/backend && PYTHONPATH=/opt/limbic .venv/bin/python3 \
  scripts/log_activity.py manual_action 'Bookify intro' \
  --detail '{\"book\":\"{book}\",\"top\":25}'"

Unresolved follow-ups

  • Per-paragraph translation inside ingest is implemented (per-paragraph Sonnet CLI calls) but still can fail on the largest paragraphs >2500 chars. The in-session direct-write pattern is the reliable fallback. Possible fix: split paragraph into halves when >2K chars, align halves separately.
  • POS tagging is inconsistent — the initial gloss batch tagged طَوَّقَ as "passive participle" (it's a verb) because the surface form present in the text was the passive participle. Glossing per-surface vs. per-lemma is the root cause. Could be fixed by re-running quality gates with grammar enrichment (already done on server for the 19 new lemmas).
  • رَأَى glossed as "I saw" instead of "to see" — same per-surface issue. Already had an existing lemma #1260; just didn't get imported fresh.
  • Paged.js for proper CSS3 footnotes — still the right long-term answer for a third format; WeasyPrint's constraint solver gives up beyond ~100 unique floats. Not attempted this session.
  • Automated vocabulary-import CLI with a named format flag (e.g., introduce --format csv <file>) so future books don't always have to be bookify-ingested first.
